## 🧾 1. 项目信息

> 📌 **项目名称**：基于 LangChain v1 + LangGraph 的多智能体出行助手  
> 🧩 **系统架构**：`fetch_user_info_node -> supervisor -> specialist agent -> tool`  
> 👥 **核心智能体**：
>
> - `supervisor_agent`
> - `flight_booking_agent`
> - `hotel_booking_agent`
> - `car_rental_booking_agent`
> - `trip_booking_agent`
> - `research_agent`

---

## 🚀 2. 测试范围

本文档用于说明：在将旧版 LangChain / LangGraph 单智能体或旧版多智能体原型迁移到最新版 LangChain v1 + LangGraph 多智能体架构后，系统当前阶段的核心测试步骤与验证口径。

当前测试主要覆盖以下能力：

- `fetch_user_info_node` 用户航班背景注入
- `supervisor_agent` 路由正确性
- `flight_booking_agent` 的查询、搜索、改签等能力
- 同一 `thread_id` 下的多轮上下文记忆
- `hotel_booking_agent/ car_rental_booking_agent / trip_booking_agent/ research_agent` 智能体的基础路由与查询行为
- 正例与负例场景下的响应稳定性

---

## ⚙️ 3. 测试环境

### 🧩 3.1 运行前提

- 多智能体图已经成功构建并编译
- 已启用 checkpointer
- 多轮对话测试时复用同一个 `thread_id`
- 测试数据库已经完成时间平移，航班分布在当前时间之后
- `context` 中使用数据库里真实存在的 `passenger_id`

### 🗂️ 3.2 当前图主流程

```text
START
  -> fetch_user_info_node
  -> supervisor_agent
  -> specialist agent
  -> END
```

### 📌 3.3 当前测试上下文示例

```python
context = {
    "passenger_id": "8252 507584",
}
```

## 🧪 4. 测试用例

### 4.1 航班智能体测试

#### ✅ 4.1.1 查询当前用户的机票信息

**测试输入**
```text
我想要查看一下我的机票信息
```

**测试目标**
- 验证 `fetch_user_info_node` 能否先注入用户当前航班背景
- 验证 `supervisor_agent` 是否能正确路由到 `flight_booking_agent`
- 验证 `flight_booking_agent` 是否能返回当前机票信息

**预期结果**
- 返回至少 1 条当前用户机票/航班信息
- 返回内容中应包含：
  - 机票号
  - 预订编号
  - 航班号 / flight_id
  - 出发机场
  - 到达机场
  - 计划起飞时间
  - 计划到达时间
  - 座位号
  - 舱位条件

---

#### ✅ 4.1.2 多轮记忆测试：不重复票号继续追问

**测试输入**
```text
我机票的航班号和座位号，起飞机场各是什么？
```

**测试目标**
- 验证同一个 `thread_id` 下，多轮短期记忆是否有效
- 验证系统是否能直接利用上一轮上下文，而不要求用户重复输入票号

**预期结果**
- 系统能够直接返回：
  - 航班号
  - 座位号
  - 起飞机场
- 不要求再次补充 `ticket_no` 或 `flight_id`

---

#### ✅ 4.1.3 查询近期从指定机场起飞的航班

**测试输入**
```text
近期从 BRU 起飞的航班都有哪些？
```

**测试目标**
- 验证 `supervisor_agent` 是否正确路由到 `flight_booking_agent`
- 验证 `flight_booking_agent` 是否调用 `search_flights`
- 验证按出发机场查询的基础能力是否正常

**预期结果**
- 返回若干从 `BRU` 起飞的航班
- 返回结果应至少包含：
  - 航班号
  - 出发机场
  - 到达机场
  - 起飞时间

---

#### ✅ 4.1.4 联合条件查询：出发机场 + 到达机场

**测试输入**
```text
帮我查询一下近期从 BRU 飞往 OSL 的航班
```

**测试目标**
- 验证 `search_flights` 是否同时吃到：
  - `departure_airport = BRU`
  - `arrival_airport = OSL`

**预期结果**
- 返回结果应明显少于“只查询 BRU 起飞”的结果
- 返回结果应满足：
  - 出发机场为 `BRU`
  - 到达机场为 `OSL`

---

#### ✅ 4.1.5 正常改签：将机票改签到有效未来航班

**测试输入**
```text
帮我把票号 9880005432001004 改签到 flight_id 为 999999 的航班
```

**前置条件**

- `flight_id = 999999` 在数据库中真实存在
- 目标航班时间位于未来
- 当前机票属于 `context["passenger_id"]` 对应用户
- 改签工具的时间解析已修正

**测试目标**
- 验证 `flight_booking_agent` 是否能正确调用改签工具
- 验证数据库写操作链路是否正常

**预期结果**
- 系统返回明确的改签成功信息
- 数据库中的 `ticket_flights` 应更新为新的 `flight_id`

---

#### ❌ 4.1.6 负例：把 `flight_no` 当成 `flight_id` 使用

**测试输入**
```text
帮我把票号 9880005432001004 改签到 flight_id 为 TG0009 的航班
```

**测试目标**
- 验证系统是否能区分：
  - `flight_id`
  - `flight_no`

**预期结果**
- 系统明确提示未找到对应航班信息
- 不应产生任何实际修改

---

#### ❌ 4.1.7 负例：当前用户无有效航班信息

**测试输入**
```text
我想看一下我的航班信息
```

**前置条件**
- 使用一个数据库中没有有效票务关联的 `passenger_id`

**测试目标**
- 验证无有效用户数据时，系统是否稳定返回空结果
- 验证不会继续进入无意义写操作

**预期结果**
- 系统返回：
  - `未查询到当前用户的航班信息`
  - 或 `未查询到您的航班信息`
- 不产生异常堆栈暴露

---

#### ❌ 4.1.8 负例：查询无匹配航班

**测试输入**
```text
帮我查询一下近期从 BRU 飞往 PEK 的航班
```

**测试目标**
- 验证 `search_flights` 无结果时的稳定性
- 验证空结果兜底是否正常

**预期结果**
- 系统返回：
  - `未查询到符合条件的航班信息`
- 不应出现胡乱编造的航班结果

## 📌 5. 建议测试顺序

为了减少排障复杂度，建议按如下顺序执行：

1. 查询当前用户机票信息
2. 多轮追问当前机票信息
3. 查询从指定机场起飞的航班
4. 查询指定出发地 + 到达地的航班
5. 查询无匹配航班（负例）
6. 执行有效改签
7. 执行无效改签（把 `flight_no` 当成 `flight_id`）
8. 酒店查询正例 / 负例
9. 租车查询正例 / 负例
10. 旅行推荐正例 / 负例
11. 联网搜索正例 / 负例
